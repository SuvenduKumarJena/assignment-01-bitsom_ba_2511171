## Database Recommendation

For a healthcare patient system, I would definitely recommend MySQL over MongoDB. Patient records need to be strictly organized and connected—like linking a patient to their exact doctor, medical history, and prescriptions. MySQL uses ACID properties. This means every database transaction is completely secure and finishes entirely, or it safely cancels itself. This stops any half-saved or mixed-up patient records from happening.

On the other hand, MongoDB uses the BASE model. It has something called "eventual consistency," which means data might take a split second to update across the whole system. In healthcare, this is a very bad idea because a doctor needs the exact, up-to-date chart right away during an emergency. Following the CAP theorem, a medical system has to pick Consistency and Partition Tolerance over Availability. It is much better for the system to pause or delay a request than to show a doctor the wrong medicine information.

If we need to add a fraud detection module, my answer changes for that specific part. Fraud detection means checking thousands of logins, locations, and actions every second to spot weird behavior. MySQL would get too slow and overloaded trying to process all that messy, unstructured data perfectly. For the fraud module, MongoDB is a much better choice. Under the CAP theorem, it can focus on Availability to handle the massive rush of incoming data without crashing the main application. So, I would keep MySQL for the main patient data to keep it safe, and use MongoDB just to handle the heavy traffic of the fraud detection system.
